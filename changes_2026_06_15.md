# Changes — 2026-06-15

All changes are in `src/bouncing_ball_task/bouncing_ball.py` and `src/bouncing_ball_task/defaults.py`.

---

## 1. Forced shape changes (`forced_shape_changes`)

**Motivation:** velocity has `forced_velocity_resamples` and color has `forced_color_changes` — shape had no equivalent.

**What was added:**

| Location | Addition |
|---|---|
| `__init__` signature | `forced_shape_changes: Optional[list[int]] = None` |
| `__init__` body | `self.forced_shape_changes` (list) and `self.forced_shape_changes_array` (bool array `(sequence_length, batch_size)`) |
| `bouncing_ball_sequence` | `forced_shape_changes_array[t]` OR'd into the shape change condition |

The shape change condition in `bouncing_ball_sequence` now reads:

```python
shape_changes_random = np.logical_or(
    rand_for_shape[t] <= self.probability_shape_change,
    self.forced_shape_changes_array[t],
)
```

This exactly mirrors the forced velocity resample and forced color change patterns. Bounce logic is untouched.

---

## 2. Shape-triggered color changes (`pccosc` and `pccovasc`)

**Motivation:** the same way a velocity change can trigger a color change, a shape change should be able to trigger one. Two new probability parameters cover the four combinations of (vel changed, shape changed).

### New parameters

| Parameter | Short name | Default | Meaning |
|---|---|---|---|
| `probability_color_change_on_shape_change` | `pccosc` | `0.0` | P(color changes \| shape changed, vel NOT changed) |
| `probability_color_change_on_velocity_and_shape_change` | `pccovasc` | `1.0` | P(color changes \| shape changed AND vel changed) |

The existing parameters cover the "shape NOT changed" cases and are unchanged in meaning:

| Parameter | Short name | Covers |
|---|---|---|
| `probability_color_change_on_velocity_change` | `pccovc` | vel changed, shape NOT changed |
| `probability_color_change_no_velocity_change` | `pccnvc` | vel NOT changed, shape NOT changed |

### The full 2×2 probability table (per batch element, per timestep)

|  | vel NOT changed | vel changed |
|---|---|---|
| **shape NOT changed** | `pccnvc` | `pccovc` |
| **shape changed** | `pccosc` | `pccovasc` |

### Default values rationale

- `pccosc = 0.0`: shape changes don't affect color unless explicitly configured — preserves existing behavior.
- `pccovasc = 1.0`: matches the existing `pccovc` default, so simultaneous vel+shape changes behave the same as vel-only changes by default.

### Where they were threaded through

| Location | Change |
|---|---|
| `__init__` signature | Two new `Union[float, Callable]` parameters added after `probability_color_change_on_velocity_change` |
| `__init__` body | Initialized to `None`; passed into `_set_callables` |
| `_set_callables` | Two new callable storage attributes added |
| `resample_change_probabilities` | Full callable/float resolution block added for both, following the same pattern as `pccovc`/`pccnvc` |
| `bouncing_ball_sequence` | Shape block **moved before** color block (see §3); color probability replaced with `effective_prob` |

### Implementation in `bouncing_ball_sequence`

The fixed `probability_color_change_given_velocity` lookup was replaced with a per-element `effective_prob` array built after shape changes are known:

```python
effective_prob = np.stack(
    [
        np.where(shape_changes_random, pccovasc, pccovc),  # col 0: vel changed
        np.where(shape_changes_random, pccosc,   pccnvc),  # col 1: vel not changed
    ],
    axis=-1,
)  # shape (batch_size, 2)

color_changes_combined = np.logical_and(
    velocity_change_nochange,
    rand_for_color[t] <= effective_prob,
)
```

`np.where` broadcasts correctly for both scalar and per-batch-element probabilities.

## 3. Shape block moved before color block in `bouncing_ball_sequence`

**Motivation:** `effective_prob` (§2) requires `shape_changes_random` to be known before the color change probability is computed.

Previously the order was: velocity → color → shape → step forward.
Now it is: velocity → **shape** → color → step forward.

The shape computation itself is unchanged; only its position in the loop moved.

---

## 4. Separate cooldown for shape-triggered color changes (`min_t_color_change_after_shape_change`)

**Motivation:** after a shape-triggered color change (pccosc path — vel NOT changed, shape changed), the cooldown should be configurable independently from purely random color changes (pccnvc path). Previously both went through `min_t_color_change_after_random`.

### New parameter

| Parameter | Default (`__init__`) | Default (`defaults.py`) |
|---|---|---|
| `min_t_color_change_after_shape_change` | `5` | `15` |

### Implementation

A `shape_triggered_color_change_array` (bool, shape `(sequence_length + max_delay, batch_size)`) is pre-allocated in `bouncing_ball_sequence`. It is written at the same delayed timestep as `color_change_array`, recording which col-1 color changes were caused by a shape change:

```python
shape_triggered_color_change_array[
    self.color_change_target_indices[t, 1], :
] = np.logical_and(color_changes_combined[:, 1], shape_changes_random)
```

The existing single col-1 cooldown is then split at read time:

```python
# Shape-triggered col-1 color changes
rand_for_color[
    t+1 : t+min_t_color_change_after_shape_change+1,
    np.logical_and(color_change_array[t, :, 1], shape_triggered_color_change_array[t]),
    1
] = 1.0

# Purely random col-1 color changes (pccnvc)
rand_for_color[
    t+1 : t+min_t_color_change_after_random+1,
    np.logical_and(color_change_array[t, :, 1], ~shape_triggered_color_change_array[t]),
    1
] = 1.0
```

Writing at the delayed timestep (not at t) means this attribution is correct even when `color_change_random_delay > 0`.

The cross-cooldown (`min_t_bounce_color_change_after_random`, which suppresses col-0 after any col-1 change) is unchanged.
